# Errors

IPProxy APIs use HTTP status codes plus a JSON response body. Business failures usually include a non-zero `code` and a Chinese `message` or `msg`.

## Common formats

```json
{
  "code": 401,
  "message": "未提供认证令牌",
  "data": {}
}
```

```json
{
  "code": 400,
  "msg": "目标用户余额不足",
  "data": {}
}
```

FastAPI validation errors may use the standard `detail` field:

```json
{
  "detail": [
    {
      "loc": ["body", "duration"],
      "msg": "field required",
      "type": "value_error.missing"
    }
  ]
}
```

## Common errors

| HTTP / Code | Message | Meaning | Action |
| --- | --- | --- | --- |
| `401` | `未提供认证令牌` | Missing Bearer token. | Send `Authorization: Bearer <token>`. |
| `401` | `无效的认证令牌` | Token is invalid or expired. | Log in again. |
| `401` | `邮箱或密码错误` | Login failed. | Check username/password. |
| `403` | `没有创建用户的权限` | Current user cannot create users. | Use admin/agent account. |
| `403` | `只能为下级用户创建订单` | Agent attempted to order for unrelated user. | Use a sub-user under that agent. |
| `400` | `目标用户余额不足` | Balance is lower than order amount. | Recharge or lower order amount. |
| `404` | `用户不存在` | Target user does not exist. | Check `userId`. |
| `500` | `创建订单失败` | Backend error during order creation. | Retry after checking request fields; contact support if persistent. |

## Retry guidance

- Do not retry validation, permission, or insufficient-balance errors without changing the request.
- Re-login and retry once for token errors.
- Retry network or 5xx failures with exponential backoff.
- For order creation, avoid blind repeated retries. Query order lists first to confirm whether an order was already created.
