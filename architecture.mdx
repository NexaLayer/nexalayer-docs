# Architecture

IPProxy public APIs are served by the IPProxy backend. There is no separate agent session gateway in the public contract; clients call IPProxy order, product, user, balance, and transaction APIs directly.

```text
Client / Partner System / Fingerprint Browser
        |
        | HTTPS + Bearer token
        v
https://api.wanyuagent.com/api
        |
        v
IPProxy Backend
  - Auth and user permissions
  - Product inventory and pricing
  - Dynamic and static order creation
  - Static proxy instances
  - Balance and transaction records
  - Supplier integrations where needed
```

## Responsibility split

| Area | IPProxy API responsibility |
| --- | --- |
| Authentication | Login and JWT Bearer validation. |
| Users | Admin/agent sub-user creation and permission checks. |
| Products | Product, region, inventory, project, and CIDR data. |
| Orders | Dynamic/static order creation, listing, renewal, and release. |
| Billing | Balance, consumption records, recharge/adjustment records. |
| Proxy access | Static instance credentials and whitelist management. |

## Request flow

1. Client logs in with `POST /auth/login`.
2. Client stores `data.token` and sends it as `Authorization: Bearer <token>`.
3. Client queries products and balance.
4. Client creates an order for a permitted `userId`.
5. Client queries order list/details to retrieve proxy instances and reconcile status.
6. Client queries transactions for billing audit.

## Domain

Use the production API domain for both normal server integrations and fingerprint browser integrations:

```text
https://api.wanyuagent.com/api
```
